You *can* do this on Solana with Irys; they support SOL/USDC on Solana and provide a Solana-specific SDK package.

***

## Does Irys support Solana?

- Irys lists **Solana (SOL)** and **USDC on Solana** as fully supported tokens for funding uploads.[^2_1]
- The SDK has dedicated Solana clients: `@irys/upload-solana` (Node) and `@irys/web-upload-solana` (browser), plus Solana helpers in their JS SDK.[^2_2][^2_3][^2_4]

So your “Git3‑style” storage layer can stay on Irys while payments and identity are Solana-native.

***

## How to use Irys from Solana

To adapt the ETH quickstart you linked to, you basically swap the payment client:

1. **Install Solana uploader**
```bash
npm install @irys/upload @irys/upload-solana
```

2. **Configure uploader with a Solana keypair**

Node/TS style (backend or GitHub Action worker):

```ts
import { Uploader } from "@irys/upload";
import { Solana } from "@irys/upload-solana";
import fs from "fs";

const getIrysUploader = async () => {
  const keypair = JSON.parse(fs.readFileSync("./.env.keypair", "utf8"));
  const irysUploader = await Uploader(Solana)
    .withWallet(keypair)
    .withRpc("https://api.devnet.solana.com");
  return irysUploader;
};
```

This pattern matches the official Solana guide.[^2_5][^2_2]

3. **Fund and upload (same API as ETH)**
```ts
const fundAccount = async () => {
  const irys = await getIrysUploader();
  const fundTx = await irys.fund(irys.utils.toAtomic(0.05));
  console.log("Funded", irys.utils.fromAtomic(fundTx.quantity), irys.token);
};

const uploadData = async () => {
  const irys = await getIrysUploader();
  const receipt = await irys.upload("hirys world.");
  console.log(`https://gateway.irys.xyz/${receipt.id}`);
};

const uploadFolder = async () => {
  const irys = await getIrysUploader();
  const receipt = await irys.uploadFolder("./git-bundles", {
    indexFile: "",
    batchSize: 50,
    keepDeleted: false,
  });
  console.log("Manifest ID", receipt.id);
};
```

Same API as the ETH example you saw, just with `Solana` instead of `Ethereum`.[^2_6][^2_7][^2_2][^2_5]

***

## Wiring this into a Solana-native “Git3”

Conceptually:

- **Identity / ownership**
    - Use a Solana **SPL token / NFT** (Metaplex standard) as the “repo asset”.[^2_8]
    - The on-chain metadata stores the latest Irys manifest ID (or an index of manifest IDs).
- **Storage**
    - GitHub Action builds a Git bundle / folder and posts to your backend.
    - Backend uses `@irys/upload-solana` to upload to Irys, returns manifest/tx ID.[^2_2][^2_5]
    - Backend or client sends a transaction to your Solana program to update the repo NFT’s metadata with that ID.
- **Payments / monetization**
    - Charge in SOL or USDC‑SOL for:
        - Repo access (clone/download endpoint).
        - Agent actions (read file, run tests, etc.).[^2_1][^2_5][^2_8]
    - You can combine this with x402 (HTTP 402) at the API layer while still settling payments on Solana.

So the only change from the ETH example is: use Solana wallet + RPC + Solana uploader, and make the repo asset + monetization logic live in a Solana program / SPL NFT instead of EVM.

Short answer: yes, you can support **both Solana and EVM** with one product, if you separate concerns cleanly: Irys as a shared data layer, chain‑specific “RepoNFT” contracts/programs, and a chain‑agnostic x402 paywall.

***

## 1. Core architecture (chain‑agnostic)

Use Irys as the single **storage + provenance** layer for all repos:

- Upload Git bundles / folders via Irys SDK (same `upload` / `uploadFolder` API for all chains).[^3_1][^3_2]
- Fund uploads in the chain/token the user chooses (SOL or EVM tokens are both supported via `@irys/upload-ethereum` and `@irys/upload-solana`).[^3_3][^3_4]
- Always get back an Irys `id` (manifest or data id), which is the canonical pointer to the repo snapshot.[^3_2]

This keeps your “Git3‑like” logic identical regardless of whether the user is an EVM or Solana dev.

***

## 2. Chain-specific ownership layer

Have separate “repo asset” implementations, but identical semantics:

- **EVM side (Ethereum/Base/Polygon/etc.)**
    - Standard ERC‑721 contract `RepoRegistryEVM` where:
        - `tokenId` identifies a repo.
        - Metadata URI includes the latest Irys manifest id.
    - Owner = EVM address; used for pricing and revenue share.
- **Solana side**
    - Solana program `RepoRegistrySol` managing one **SPL NFT / pNFT** per repo.[^3_5][^3_6]
    - PDA or metadata account stores:
        - Repo id, current Irys manifest id, pricing config.
    - Owner = Solana public key.

Key: both sides only store *pointers* to Irys, so your data architecture is unified; only the ownership contract/program differs.

***

## 3. Payments and paywall (x402 + chain choice)

Use **x402** as the HTTP‑level payment standard, but allow payment on either EVM or Solana:

- x402 is explicitly **blockchain‑agnostic** and already supports multiple chains, including Solana and EVM networks.[^3_7][^3_8][^3_9][^3_10]
- Your API flow:

1. Client/agent calls `GET /repos/:id/manifest`.
2. If free → return Irys gateway URL directly.
3. If paid → respond `402 Payment Required` with x402 payment instructions that list:
        - Supported chains (e.g., `base`, `ethereum`, `solana`).
        - Accepted tokens (e.g., USDC on Base, USDC on Solana).[^3_8][^3_9][^3_10][^3_7]
4. Client pays via chosen chain; facilitator confirms; you re‑serve with the Irys manifest/gateway URL.

This gives you one **unified paywall** while still settling on the user’s preferred chain.

***

## 4. Backend plan (one codebase, multi‑chain)

Backend (Node/TS) responsibilities:

- **GitHub integration**
    - OAuth, repo list.
    - Webhook / GitHub Action callback that:
        - Receives repo bundle.
        - Calls Irys `uploadFolder` to store it.[^3_1][^3_2]
- **Irys client factory**
    - `getIrys(chain)`:
        - For EVM: `Uploader(EthereumLike).withWallet(privateKey)` or `new Irys({ token: "matic", ... })`.[^3_4][^3_3]
        - For Solana: `Uploader(Solana).withWallet(solKeypair)` or `new Irys({ token: "solana", ... })`.[^3_3][^3_4]
    - Same upload code for both; only token + key differ.
- **Repo registry integration**
    - If user chooses EVM:
        - Call `RepoRegistryEVM.mintOrUpdate(repoId, irysId, price, ...)`.
    - If user chooses Solana:
        - Submit a Solana transaction to your `RepoRegistrySol` instruction with `repoId`, `irysId`, `price`.
- **x402 gateway**
    - Implement the resource server:
        - Read repo metadata (chain, owner, price).
        - If paid, respond 402 with x402 spec payload, listing supported chains.[^3_9][^3_10][^3_7][^3_8]
        - After facilitator callback / verification, return the Irys URL or stream data.

### 1. Open source developers \& creators

**Problem**: GitHub/GitLab are centralized and fragile—you can get banned, repos deleted, or lose access. No way to monetize your open source work beyond donations/sponsors.

**Git3‑like solution**:

- Mirror private repos on‑chain with cryptographic proof of existence/history.
- Enterprises pay for access or verification services.
**Value**: Compliance + provable provenance without changing workflows.


### 3. AI agent developers

**Problem**: AI agents need to read/run code autonomously, but can't pay for access easily. Current APIs don't have micropayments or on‑chain verification.
**Git3‑like solution**:

- Agents call your x402‑protected APIs: "read file X", "run tests", "summarize diff".

**Value**: Economic access to verified code for autonomous dev tools.
